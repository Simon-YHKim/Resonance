## CHAPTER 5. 방구석/실내 플레이 모드 — 2D 타일맵 시스템

### 5.1 설계 원칙

| 원칙 | 내용 |
|---|---|
| **목적** | 비·야간·이동 불가 상황에서도 끊김 없는 플레이 보장. 그러나 GPS 모드를 대체하지 않음. |
| **분리** | 가상 모드 ↔ GPS 모드는 **완전 격리**. 캐릭터/인벤토리는 공유, 활동 좌표·드롭 회수는 모드 내에서만 유효. |
| **의도된 한계** | 보상 80%, 운명 등급 장비 드롭률 50%, 일부 월드 보스·지역 한정 몹 미출현. 가상 모드만으로 “끝판” 도달 불가. |
| **시각** | 텍스트 + 2D 톱다운 픽셀 타일맵. 채팅창은 화면 하단 30%, 타일맵은 상단 70%. |

### 5.2 게임 엔진 선정 — **Phaser 3 단독 추천**

조사 결과 1순위는 **Phaser 3 (현재 v3.90+ / 2026)** 으로 명확하다.

| 항목 | Phaser 3 ⭐ | PixiJS | Excalibur.js | Cocos Creator | Godot HTML5 |
|---|---|---|---|---|---|
| 종류 | 게임 프레임워크(올인원) | 렌더러만 | TS 기반 OOP 엔진 | 통합 IDE | 데스크톱 엔진의 웹 export |
| 번들 크기 | ~500KB | ~200KB | ~300KB | 1~2MB | 8MB+ (빈 프로젝트도) |
| Tiled JSON 직접 로드 | **네이티브 지원** (`load.tilemapTiledJSON`) | 직접 구현 필요 | 플러그인 | 자체 포맷 | 자체 포맷 |
| 모바일 터치/제스처 | 내장 (Pointer, gesture plugin) | 직접 구현 | 내장 | 내장 | 보통 |
| 학습 자료 | 1,700+ 공식 예제, GitHub 36k★ | 적당 | 적음 | 한국어 문서 적음 | 한국어 문서 적음 |
| **LLM 코드 생성 친화도** | **최상** (Claude/GPT 학습 데이터 풍부) | 상 | 중 | 중 | 하 |
| Capacitor + React 통합 | iframe/직접 mount 모두 안정 | 안정 | 가능 | 권장 안 됨 | 권장 안 됨 |
| 라이선스 | MIT | MIT | BSD-2 | 자체(상업 OK) | MIT |

**Phaser 3을 1순위로 선택하는 결정적 이유 3가지**

1. **LLM 코드 생성 정확도**: Vibe 코딩의 핵심은 “Claude/Sonnet에 ‘플레이어 이동 + 충돌 + Tiled 맵 로드’를 시켰을 때 한 번에 동작하는가”다. Phaser는 학습 데이터가 압도적으로 많아 hallucination이 가장 적다.
2. **올인원**: 물리(Arcade), 카메라, 트윈, 사운드, 입력, 씬 매니저, Tiled 파서가 모두 내장. PixiJS는 이 모든 것을 직접 붙여야 하며 1인 개발자에겐 비현실적.
3. **Capacitor + React 안정성**: React 라우팅 안에서 `<div id="phaser-root">`에 mount 하는 패턴이 표준화되어 있어 모바일 빌드에서도 문제가 적다.

**보조 결정**:
- **TypeScript 사용 권장** (Phaser는 공식 d.ts 제공). LLM이 타입을 보고 더 정확한 코드를 만든다.
- 렌더링은 `WEBGL` 우선, 자동 Canvas fallback. iOS Safari 구버전 대응.

### 5.3 타일맵 디자인 도구 — **Tiled Map Editor 1순위**

| 도구 | 추천도 | 이유 |
|---|---|---|
| **Tiled** ⭐ | **1순위** | 무료, 표준, Phaser가 JSON을 즉시 파싱. 한국어 자료 풍부. |
| LDtk | 2순위 | UI는 더 모던하고 Auto-Layer가 강력하지만, Phaser 로드는 별도 파서/플러그인 필요. 단기 생산성은 Tiled가 우위. |
| Sprite Fusion | 보조 | 웹 기반, 빠른 프로토타이핑용 |

**워크플로우 표준안**:

```
[Tiled에서 .tmj(JSON) export]
   ↓
[/public/maps/town_01.json 에 배치]
   ↓
this.load.tilemapTiledJSON('town_01', '/maps/town_01.json')
this.load.image('tiles_urban', '/assets/kenney_urban.png')
   ↓
const map = this.make.tilemap({ key: 'town_01' })
const tiles = map.addTilesetImage('urban', 'tiles_urban')
const ground = map.createLayer('Ground', tiles)
const collide = map.createLayer('Collide', tiles).setCollisionByProperty({ collides: true })
```

> **주의**: Phaser 파서는 “Collection of Images” 타일셋과 무한 맵을 지원하지 않는다. Tiled 생성 시 **단일 이미지 시트 기반, Fixed Size**로 만들어야 한다.

### 5.5 가상 맵 구조 — **하이브리드 (정적 허브 + 절차 던전)** 채택

**무한 절차 생성 vs 정적 맵의 비용 분석**

| 방식 | LLM 호출 | 콘텐츠 다양성 | 1인 운영 부담 | 결론 |
|---|---|---|---|---|
| 완전 정적 맵 | 0 | 낮음 (질림) | 맵 수십 개 그려야 | ✗ |
| 매 입장마다 LLM이 맵 생성 | 매우 높음 (₩100+/입장) | 최상 | 비용 폭발 | ✗ |
| **허브(정적) + 던전(시드 기반 절차)** ⭐ | 던전당 0회, 몹 묘사만 LLM | 높음 | 적정 | **채택** |

**가상 맵 구성안**

```
        [북: 서리의 숲 던전]
              ↑
[서: 망각의 폐허] ← [중앙 허브: 방구석 마을] → [동: 데이터 카타콤브]
              ↓
        [남: 심해의 회랑 던전]
```

- **중앙 허브 “방구석 마을”** (정적, 64×64 타일): NPC 상점·길드·게시판·스프린트 모드 입구·복귀 포털.
- **4방위 던전**: BSP(Binary Space Partitioning) 또는 Drunkard Walk 알고리즘으로 시드 기반 절차 생성. 같은 시드는 같은 맵 → 일별 시드(`YYYY-MM-DD`) 사용 시 “오늘의 던전”이 모든 유저에게 공유 = 길드/리더보드 가능.
- **층수**: 던전당 5~10층. 깊을수록 보상↑ 사망 시 회수 어려움↑.
- 절차 생성 코드는 Phaser 안에서 **JS로 100% 처리** → LLM 호출 0. 생성된 방에 “몹 슬롯”만 할당하고, 몹 묘사 텍스트만 LLM(Flash-Lite)으로 생성.

**가상 맵 몹 풀** (지역 특성 없음 → 다른 변별력 필요)

| 변별 축 | 예 |
|---|---|
| 던전 테마 | 숲 = 자연 변종 / 폐허 = 언데드 / 카타콤브 = 디지털 글리치 / 심해 = 크리쳐 |
| 층 깊이 | 1~3F 일반, 4~7F 정예, 8~10F 보스급 |
| 시간대(현실 시각 연동) | 새벽/낮/저녁/밤마다 등장 풀 변화 |
| 주간 모디파이어 | “이번 주 데이터 카타콤브 = 화염 면역” 식 로테이션 |

**GPS 모드 대비 페널티 매트릭스**

| 항목 | GPS 모드 | 가상 모드 |
|---|---|---|
| 골드/경험치 | 100% | **80%** |
| 일반 장비 드롭률 | 100% | 90% |
| 운명(전설) 등급 드롭률 | 100% | **50%** |
| 월드 보스 | 출현 | 미출현 |
| 지역 한정 몹 | 출현 | 미출현 |
| 이벤트 “현장 부스트” | 적용 | 비적용 |
| 스프린트 모드 입장 | 가능 | 가능 (동일) |

## CHAPTER 6. 세키로식 사망/금화 시스템 — 한국형 변형

### 6.1 원작 메커니즘 정밀 분석

세키로 자체에는 **‘사망 위치에 골드를 떨어뜨리고 회수’**라는 시스템이 없다. 그것은 **다크 소울/엘든 링** 계열이다. 사용자의 결정사항은 두 게임을 결합한 형태이므로 정확히 정리한다.

| 항목 | 세키로 (Sekiro) | 엘든 링/소울 시리즈 |
|---|---|---|
| 1차 부활 | **현장 부활(Resurrection)** — 같은 자리에서 즉시. 1번 충전, 적 처치 시 충전. | 없음 |
| 2차 사망(진사) 페널티 | 보유 **센(돈) 50% + EXP 50% 영구 소실**. 회수 불가. | 보유 **룬/소울 100%를 사망 위치에 드롭**. 회수 가능. |
| 회수 메커니즘 | 없음 | **떨어진 룬 위로 가면 회수**. 회수 전 또 죽으면 **영구 소실**. |
| 보호 장치 | **Unseen Aid** (기본 30% 확률로 사망 페널티 무효) | 없음 (아이템으로 일부 보존 가능) |
| 사회적/세계 영향 | **Dragonrot** — 죽음 누적 시 NPC가 병에 걸려 퀘스트 정지. Dragon’s Blood Droplet으로 해제. | 없음 |
| 마지막 저장지점 | **조각상(Sculptor’s Idol)** | 사망 지점 / 마지막 화톳불 선택 불가, 화톳불 강제 |

**디자인 심리학 핵심 인사이트**:
- 소울 시리즈의 *“두 번 죽으면 영구 손실”* 은 **위험-회수의 결정 심리**(prospect theory)를 극대화한다. 플레이어는 회수를 위해 무리하다 더 큰 손실을 본다 → 강한 기억 형성.
- 세키로의 Dragonrot은 *“네 죽음이 세계를 더럽힌다”* 는 서사적 페널티로, 숫자 손실보다 죄책감을 자극한다.

### 6.2 본 게임 통합 룰 — “**한국형 죽음의 무게**” 시스템

> 사용자 결정을 그대로 반영하되, GPS라는 물리적 제약을 고려하여 변형.

#### 6.2.1 사망 단계 및 처리

| 단계 | 트리거 | 결과 |
|---|---|---|
| **0차 (위기)** | HP 0 도달 시 | LLM 사망 묘사 + 부활 토큰 보유 시 1차 부활 옵션 노출. |
| **1차 사망** | 부활 토큰 사용 또는 첫 사망 | **선택지 제공**: ⓐ 현장 부활 (HP 50%, **5분간 능력치 -20% 디버프**) / ⓑ 마지막 저장지점 부활 (디버프 없음, 위치만 이동) |
| **2차 사망** | 1차 부활 후 다시 사망 / 부활 토큰 부재 | 마지막 저장지점 강제 부활 + **보유 골드 100% 드롭, 잔고 0**. 사망 좌표에 ‘죽음의 표식’ 생성. |
| **회수** | 표식 위치 도달 + 상호작용 | 골드 100% 회수. 표식 소멸. |
| **영구 소실** | 표식 회수 전 또 사망 | 이전 표식 골드 **영구 소실**. 새 표식 생성. |

**Why 디자인 결정**:
- 모바일·LLM 게임은 평균 세션이 짧다(5~15분). 100% 골드 손실은 가혹하지만 **자주 사망하지 않으면 체감 빈도는 낮다** → 한 번 죽었을 때의 ‘진지함’을 위해 가혹함 유지.
- 1차 부활의 디버프는 “세키로 부활” 의 LLM 맥락 변형. LLM은 부활 직후 **“기적적 회복 묘사”**를 자동 생성: *“숨이 끊어진 줄 알았다. 그러나 손끝에 닿은 차가운 공기가 너를 다시 일으켜 세웠다. 어깨가 무겁고, 시야가 흔들린다.”*

#### 6.2.2 GPS 모드 ↔ 가상 모드 격리

| 사망 발생 모드 | 표식 생성 위치 | 회수 가능 모드 |
|---|---|---|
| GPS 모드 (현실 좌표) | 카카오맵/구글맵 위 GPS 좌표 | **GPS 모드에서만** |
| 가상 모드 (2D 타일맵) | 가상 던전/마을 좌표 | **가상 모드에서만** |

이 격리는 “출장 중 GPS 사망 위치를 가상에서 회수” 같은 회피를 막는다. **‘출장 페널티’** 는 의도된 디자인 요소다.

#### 6.2.3 회수 불가 거리 보호 장치 (현실적 안전망)

GPS 모드에서 “부산 출장 중 서울 한복판에서 죽었다” 시나리오 대응:

| 안전망 | 발동 조건 | 효과 |
|---|---|---|
| **유실 청구 (Lost Wallet Claim)** | 표식 생성 후 **30일** 미회수 | 골드의 **30%**가 “저주받은 동전”으로 우편함 자동 송부. 70%는 영구 소실 (페널티는 유지). |
| **운명석 회수권 (IAP)** | 결제 ₩2,200 / 1회 | 위치 무관 즉시 회수. 단 골드의 **20% 수수료** 차감. |
| **신참 보호 (Lv ≤ 10)** | 자동 | 사망 시 골드 50%만 드롭, 50% 즉시 보존. |
| **부적 — “수호의 동전”** | 인게임 제작/획득 | 1회용. 사망 시 골드 50% 자동 보존. 슬롯 1칸 점유. |

#### 6.2.4 골드/스태미나/운명석(하드 커런시) 차등 처리

| 자원 | 사망 드롭 | 사유 |
|---|---|---|
| **골드 (소프트)** | **100%** | 위험-회수 긴장의 핵심 |
| 경험치 | 0% (보호) | 모바일 RPG에서 EXP 손실은 이탈률 급증 |
| 스태미나 | 0% (보호) | 시간 자원이라 손실 시 무력감 큼 |
| **운명석 (하드/IAP)** | **0% (절대 보호)** | 환불·법적 리스크. 한국 게임물관리위 요구사항 |
| 장비/소비 아이템 | 0% (보호) | 회수 시스템 추가 시 불공정·UX 악화 |

> **결정**: 본 게임의 “세키로식”은 **골드 단일 자원에만 적용**한다. 이 단순함이 비개발자 운영을 가능하게 한다.

#### 6.2.5 PK 정책

- **출시 시점: PvE 한정**. 타 플레이어를 죽이거나 표식을 가로채는 행위 불가.
- 사유: (1) 한국 모바일 RPG 시장에서 PK는 호불호 극단, (2) LLM 비용 폭증 위험, (3) 1인 운영자가 분쟁 처리 불가.
- 향후: 길드전·시즌 “심연의 균열” 같은 **격리된 PvP 모드**로만 도입 검토.

### 6.3 사망 위치 표시 UX

| 모드 | UX |
|---|---|
| GPS 모드 | 카카오맵 위 **붉은 십자가(💀) 마커** + 사망까지의 거리 표시. 100m 이내 진입 시 "회수" 버튼 활성화. |
| 가상 모드 | 미니맵 빨간 점, 메인 화면 표식 스프라이트(애니메이션 깜빡임), Z키/탭으로 회수. |

### 6.4 사회적 시그널 — “죽음의 발자취”

다른 유저가 사망 좌표 근처를 지나가면:

> 💀 *“여기서 누군가가 쓰러졌다. 차가운 공기가 흐른다.”* (LLM 30자 묘사, 캐싱)

- 자기 표식만 회수 가능, 타인 표식은 정보만.
- 익명: 닉네임 표시 X, “Lv.27 모험가” 정도만.
- 다크소울 “피의 흔적” 시스템의 한국형 변용. 무료 LLM 호출은 0 (정해진 30개 묘사 풀에서 랜덤).

### 6.5 6장 리스크와 대응

| 리스크 | 확률 | 대응 |
|---|---|---|
| 가혹함으로 신규 유저 이탈 | 중 | Lv ≤ 10 신참 보호 + 첫 사망 시 “이 게임의 죽음은 무겁습니다” 튜토리얼 모달 |
| GPS 회수 물리 불가능 누적 → 환불 요구 | 중-고 | 30일 유실 청구 + 운명석 회수권 IAP 명시 |

### 7.2 본 게임 스프린트 모드 사양

#### 7.2.1 4단계 난이도 매트릭스

| 난이도 | 몹 수 | 입장 비용(스태미나) | 시간 제한 | 사용 LLM | 보상 배수 | 입장 제한 |
|---|---|---|---|---|---|---|
| **하** | 10 | 50 | 10분 | Gemini 2.5 Flash-Lite | 1.0× | 무제한 |
| **중** | 15 | 100 | 12분 | Gemini 2.5 Flash | 1.5× | 무제한 |
| **상** | 20 | 200 | 15분 | Claude Haiku 4.5 / GPT-5-mini | 2.5× | 일 3회 |
| **극상** | 30 + 미니보스 1 + 최종보스 1 | 400 | 20분 | **Claude Sonnet 4.6** | 5.0× + 시즌 한정 외형 | **주 1회** |

> *모델명은 2026년 4월 기준 일반화된 명명. v1.0의 LLM 라우팅 전략과 정합.*

#### 7.2.2 몹 등장 패턴

```
[입장] → [점층 구간 1: 약한 몹 점차 강해짐] → [페이즈 전환: LLM 휴식 메시지]
       → [밀집 구간 2: 동시 다수 조우 (2~3마리 동시)]
       → [중간 미니보스 (상/극상만)]
       → [정예 구간 3: 강력한 단일 몹]
       → [최종 보스]
       → [클리어 보상 일괄 지급]
```

- **랜덤 vs 점층**: 점층(난이도 곡선) 주축, 랜덤(시드 기반)으로 몹 종 선정. → 매번 다른 경험 + 학습 가능한 패턴.
- **도중 사망 처리**: 스프린트 내부 사망은 **6장의 골드 드롭 적용 X (보호 영역)**, 단 입장료 스태미나는 환불 X. 즉, 시간/스태미나만 손실. → 도전 부담 완화.
- **재도전**: 도중 포기 시 보상 0. 클리어 시에만 일괄 보상.

#### 7.2.3 프롬프트 연마 평가 — “**효율 점수(Efficiency Score)**”

LLM이 매 입력의 효율을 자동 평가 → 별 1~5개 등급. 평가는 클리어 후 일괄(추가 LLM 호출 1회)로 비용 최소화.

| 평가 항목 | 가중치 | 측정 |
|---|---|---|
| **턴 수** | 35% | 적은 턴 = 높음. “1턴 킬” 발생 시 가산점 |
| **클리어 시간** | 25% | 빠를수록 높음 (시간 제한 대비 비율) |
| **입력 평균 길이** | 15% | 50자 이내 보너스(간결성), 200자 초과 감점(LLM 비용) |
| **창의성** | 15% | 클리어 후 평가 LLM이 입력 로그 분석 — 단조반복 vs 전략 다양성 |
| **노 데미지** | 10% | HP 손실률 |

**보상 적용 공식**:
```
최종 보상 = 기본 보상 × 난이도 배수 × (1 + 효율 점수/10)
예) 극상 클리어, 효율 ★★★★★(점수 9.5)
   = 기본 1000G × 5.0 × (1 + 0.95) = 약 9,750G
```

**도전 과제**(시즌 패스 진행도 연동):
- *“1턴 킬”* — 단일 입력으로 몹 1마리 처치 (×10 누적)
- *“간결의 검”* — 평균 입력 30자 이내로 클리어
- *“창의의 영감”* — 같은 단어 3회 미만 사용
- *“완전무결”* — 노 히트 클리어

```
  │   └ OSM Overpass API → 지역 내 amenity/shop/leisure 태그 카운트 → R2
  │
  └ 주 1회 (월요일 05:00 KST):
      └ LLM Batch (Sonnet 4.6 또는 GPT-5):
          입력: 지역명 + 위 모든 데이터 요약
          출력: JSON {지역, 정체성, 핵심 몹 5종, 트렌드 몹 3종}
          → Cloudflare KV에 저장 (key: region:{guId}:{dongId})
```

런타임에는 KV에서 읽기만 함 → 사용자별 LLM 호출 0회. 비용 통제 핵심.

### 8.4 지역별 컨셉 예시 (사전 큐레이션 시드)

| 지역 | 정체성 | 핵심 몹 5종 |
|---|---|---|
| **성수동** | 힙스터·MZ·팝업 | 셀렙 인플루언서, 팝업의 망령, 코너의 바리스타, MZ 모델의 그림자, 인스타 셔터의 영혼 |
| **강남(역삼/논현)** | 의료·법무·명품·학원 | 차가운 변호사, 성형 컨설턴트, 1타 강사, 명품의 망자, 강남의 빌딩 거인 |
| **홍대** | 인디·클럽·청춘·술 | 지하 보컬리스트, 새벽 4시의 술꾼, 골목길 버스킹의 혼, 클럽의 베이스 괴물, 헌책방의 시인 |
| **종로(인사·익선)** | 역사·한옥·노포 | 조선 시대 학자의 영혼, 노포 국밥집 주인, 한옥 처마의 까치, 인사동 화선지의 글자, 종로 시계탑 망령 |
| **여의도** | 금융·회사원·점심 | 지친 차장, 엘리베이터의 영혼, 점심시간의 햄버거 거인, 한강 바람의 영업, 9시 회의실의 그림자 |
| **이태원** | 다국적·클럽·맛집 | 케밥의 향연, 술집 골목의 야경 정령, 다국어의 혼란, 옥상 파티의 메아리 |
| **건대/성수카페거리** | 학생·노포 + 카페 | 막창 굽는 노포 할머니의 영혼, 시험기간의 좀비, 카공족 그림자 |
| **잠실/송파** | 가족·아파트·롯데 | 아파트 단지의 수호령, 놀이공원 마스코트의 변형, 한강 따라 도는 자전거 라이더 |
| **마포(연남/망원)** | 카페·산책·공원 | 경의선 숲길 토끼, 망원시장 호객의 영혼, 연남 와인의 향기 |
| **강북(우이/수유)** | 산·일상 | 북한산 등산객 영혼, 동네 서점의 학자, 수유시장의 호떡 장인 |

> 출시 시점: 서울 25개 구 × 평균 4 핫스팟 = **약 100개 지역**.
> 각 지역 5 핵심 + 3 트렌드 = **800개 몹 정의**.
> LLM 자동 생성으로 80% 처리, 1인 제작자는 20%(예 200건)만 검수/조정.

### 8.5 갱신 주기 정책

| 지역 등급 | 갱신 주기 | 트렌드 몹 비율 |
|---|---|---|
| **S급** (강남/홍대/성수/이태원 등 10개) | **주 1회** | 30~50% |
| A급 (인지도 있는 동·역세권 30개) | 격주 | 25% |
| B급 (서울 일반 동, 광역시 핫스팟 60개) | 월 1회 | 10% |
| C급 (지방 일반) | 분기 1회 | 0% (사전 큐레이션 고정) |

**핵심 원칙**: **핵심 몹 5종은 시즌 단위(3개월)로 고정**, **트렌드 몹 3종만** 위 주기로 갱신. 지역 정체성은 보호되고, 신선도는 유지.

### 8.9 데이터 신선도 vs 안정성 — 하이브리드 정책 요약

```
[지역 = 핵심 정체성 (시즌 단위 고정)] + [트렌드 표층 (주/월 갱신)]
              80%                              20%
```

이 비율이 “성수동의 본질은 갑자기 변하지 않는다”와 “요즘 뭐가 핫한가는 반영해야 한다”의 균형점.

### 8.10 8장 리스크와 대응

| 리스크 | 대응 |
|---|---|
| LLM 환각으로 비현실/부적절 몹 | 2단계 필터(생성 LLM + 검증 LLM), 1인 검수 주 1회 |
| 트렌드 너무 자주 바뀜 → 유저 “어제 봤는데 사라졌다” 혼란 | 핵심 몹 80% 고정 정책, 갱신 시 “이번 주의 변화” 패치노트 자동 생성 |
| 지방 데이터 부족 → 지역 콘텐츠 격차 | C급은 사전 큐레이션 고정, “지방 특색 시즌 이벤트” 별도 운영 |
| 비용 폭주 | 배치 API 강제, Cloudflare KV 캐싱, 월 예산 알림 |
| 한국 정치/종교/지역감정 폭발 | 시스템 프롬프트에 “정치/종교/지역감정 키워드 시 생성 거부” 룰. 사고 시 핫픽스 1시간 내 해당 지역 KV 롤백 |
